# MacParakeet v0.7.3 Telemetry Health Audit

**Snapshot:** 2026-07-17, approximately 10:20 PDT

**Release:** v0.7.3, published 2026-07-17 01:44:51 UTC

**Verdict:** Continue rollout. The release remains healthy, the historical
stats rollup remediation is deployed and verified, raw deletion remains
disabled, and the observed crash is part of a small pre-existing crash family
rather than a true one-off.

## Executive verdict

- **No rollback or emergency app hotfix is indicated.** In the first roughly
  16 hours, v0.7.3 had 181 GUI sessions, 683-689 dictation starts (the count
  advanced during the audit), 109 transcription starts, and one crashing
  session: 99.45% crash-free.
- **The crash count is one on v0.7.3, but the signature is not one-off.** The
  same normalized 11-frame SIGSEGV signature appeared four times on v0.7.2 and
  once on v0.7.3, always on macOS 26.5.2. The evidence points toward a recurring
  SQLite/CoreData/SwiftUI framework-side family, not a new v0.7.3 regression.
  The initiating app operation cannot be established from the available custom
  crash payload.
- **Raw-event retention is not operating.** The retention Worker is present in
  source but is absent from the live Cloudflare account. Production contains
  62,943 raw events older than the documented 90-day limit.
- **The stats path is too scan-heavy.** D1 reported 5.788 billion rows read in
  the preceding 24 hours. If that day were representative for a 30-day month,
  paid D1 row-read overage would be approximately **$149/month**. One uncached
  `/api/stats` request returned HTTP 500; three subsequent requests succeeded.
  The expensive five-minute snapshot regeneration is the leading hypothesis,
  not a proven root cause of that single 500.
- **The deterministic reviewer has a surface-classification bug.** It maps
  `surface IS NULL` Sparkle checks to GUI and groups by the underlying column,
  producing duplicate GUI rows and inflating GUI exposure in its report. The
  public stats endpoint correctly uses `surface = 'gui'` and is not affected by
  this specific bug.

## Crash deep dive

The v0.7.3 report was emitted at `2026-07-17T10:48:50Z`, ten seconds after the
recorded crash time. That confirms the relaunch reporter path worked. It was a
SIGSEGV on an M4 Max running macOS 26.5.2.

The production DMG was downloaded again from the GitHub release rather than
using the stale local `dist` directory. Its executable UUID is
`2F269AAE-3409-321F-B387-0F2392D734E9`, exactly matching the crash payload. The
first frame resolves to the CrashReporter signal-handler closure, which is
expected for this custom signal report. On a matching macOS 26.5.2 host, the
next useful system frame maps into SQLite (`modifyPagePointer`), followed by
SwiftUI/CoreData frames and a SQLite locking tail.

After normalizing for ASLR, the entire 11-frame signature matches reports at:

- v0.7.2: July 14 at 09:32 and 09:33 UTC, July 16 at 20:25 UTC, and July 17 at
  07:20 UTC
- v0.7.3: July 17 at 10:48 UTC

This is enough to reject “one-off” as a signature-level conclusion, but not
enough to blame a specific app call site. There is no matching production dSYM
retained with the release, and the custom payload lacks Apple exception context,
thread state, and a reliably attributable pre-signal app frame. Open a tracking
issue keyed by the normalized signature and request Apple crash logs from
affected users before changing persistence or SwiftUI code.

## Telemetry infrastructure findings

### 1. The 90-day retention promise is not being met

The source tree defines `macparakeet-telemetry-retention`, scheduled daily at
02:00 UTC, and deletes `events` rows older than 90 days. A live
`wrangler deployments list --config wrangler.telemetry-retention.toml` check
returned that the Worker does not exist on the account. The oldest production
row was dated 2026-03-13, with 62,943 rows beyond the cutoff.

This is a privacy/governance issue before it is a capacity issue. The database
is about 950 MB, roughly 9.5% of D1's current 10 GB paid per-database maximum.
However, **do not deploy the raw delete in isolation**: `/api/stats` still
computes its All Time cards, cumulative/daily history, all-time speed, per-chip
speed, and dictation-destination mix directly from raw events.

At the follow-up snapshot, deleting the expired rows without rollups would have
removed 56,819 release-shaped GUI events from the public stats data set (3.8%),
about 1,872 currently counted distinct sessions (8.1%), 4.67 million words
(5.4%), 478 hours of transcription audio, and 146 hours of dictation audio. The
public history would begin on April 18 instead of the March 13 launch-data
boundary, shrink from 127 to 91 daily buckets, and lose one historical country.
The 24-hour and 30-day operational panels would remain intact, but the page's
“All Time” and “since launch” labels would no longer be truthful.

First materialize historical daily and cumulative rollups, switch the stats
endpoint to combine immutable pre-retention rollups with the live raw window,
and verify a before/after payload comparison. Then deploy the scheduled Worker,
confirm the old-row count reaches zero, and add an alert or deploy-time check so
source presence cannot be mistaken for production operation. This sequencing
also removes the main source of the stats endpoint's raw-scan cost. See
[Cloudflare's current D1 limits](https://developers.cloudflare.com/d1/platform/limits/).

### 2. Stats regeneration is expensive and occasionally brittle

D1's 24-hour counters at snapshot were:

- 26,974 read queries
- 35,076 write queries
- 5,788,330,111 rows read
- 256,434 rows written

The stats endpoint stores one aggregate snapshot for only five minutes. When it
expires, it rebuilds a large response using many raw-event aggregate queries.
At a sustained 5.788B rows/day, the projected 30-day total is 173.65B rows. Paid
D1 includes 25B rows/month and charges $0.001 per additional million rows, so
the illustrative excess is 148.65B rows, or about $148.65/month. Writes and
storage remain within the included paid allowances. See [Cloudflare's current
D1 pricing](https://developers.cloudflare.com/d1/platform/pricing/).

Replace repeated raw scans with incremental daily/hourly rollups, lengthen the
snapshot TTL where freshness permits, and make stale snapshot serving the
fallback when regeneration fails. Measure query-level rows-read before and
after. The single observed HTTP 500 is not enough to prove that scan volume was
its cause, but it makes this a reliability issue as well as a cost issue.

### 3. The reviewer conflates Sparkle and GUI traffic

Sparkle checks intentionally have `surface IS NULL`. The deterministic reviewer
uses:

```sql
COALESCE(surface, CASE WHEN event = 'cli_operation' THEN 'cli' ELSE 'gui' END)
```

and then `GROUP BY surface`, where `surface` can bind to the underlying column
rather than the rendered alias. That produces a second apparent GUI group and
counts Sparkle device hashes as GUI sessions. Make the expression explicit in
the grouping (or group by column position), and classify `sparkle_check` as an
`update` surface rather than GUI. Add a fixture with GUI, CLI, and NULL Sparkle
rows. The deployed public stats path already uses strict GUI and CLI filters.

## Recommended order of work

1. **Land historical rollups, then deploy and verify retention.** Preserve the
   public All Time contract in aggregate form, compare the stats payload before
   and after, deploy the delete job, and verify both retention and fresh writes.
   Treat this as one urgent change because it closes the privacy gap and the
   dominant raw-scan cost together.
2. **Add stale-on-error behavior and stats health monitoring.** Track rows-read
   daily and alert on cold-regeneration failure.
3. **Create a crash-family issue, not an emergency patch.** Attach the five
   normalized signatures, exact production UUID, OS concentration, and a request
   for native Apple crash reports. Re-evaluate at 500 and 1,000 v0.7.3 sessions
   or sooner if the rate rises.
4. **Fix the deterministic reviewer's surface SQL.** Re-run the release report
   after separating Sparkle from GUI.
5. **Continue product-health follow-up without blocking rollout.** Validate the
   `mic_silent` detector against real audio gaps before Phase B recovery; split
   expected model optimization from genuinely stuck engine starts; keep LLM
   failures segmented by provider/version.
